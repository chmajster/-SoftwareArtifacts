# Contributing to SoftwareArtifacts

Thank you for your interest in contributing! The following guidelines will help you get started.

## Code of Conduct

By participating in this project you agree to be respectful and constructive in all interactions. We are committed to providing a welcoming and harassment-free experience for everyone.

## How to Contribute

### Reporting Issues

- Search [existing issues](https://github.com/chmajster/-SoftwareArtifacts/issues) before opening a new one.
- Use a clear, descriptive title.
- Describe the problem in detail, including steps to reproduce if applicable.

### Suggesting Enhancements

- Open an issue with the `enhancement` label.
- Explain your use case and why the change would benefit others.

### Submitting Pull Requests

1. **Fork** the repository and create your branch from `main`:

   ```bash
   git checkout -b feature/your-feature-name
   ```

2. **Make your changes** following the style of existing files.

3. **Commit** with a clear, descriptive message:

   ```bash
   git commit -m "Add: brief description of change"
   ```

4. **Push** your branch and open a pull request:

   ```bash
   git push origin feature/your-feature-name
   ```

5. Ensure your PR description clearly explains **what** changed and **why**.

### Commit Message Guidelines

Use the following prefixes:

| Prefix | Purpose |
|--------|---------|
| `Add:` | New artifact or feature |
| `Fix:` | Bug fix |
| `Update:` | Improvement to an existing artifact |
| `Remove:` | Deletion of an artifact |
| `Docs:` | Documentation-only change |

## Style Guide

- Keep file content general enough to be reused across projects.
- Use clear, plain English in all documentation.
- Avoid project-specific hardcoded values; use placeholders where appropriate (e.g., `<project-name>`, `<year>`).

## Review Process

All pull requests are reviewed by a maintainer. Reviews typically happen within a few business days. Please be patient and respond to feedback promptly.
